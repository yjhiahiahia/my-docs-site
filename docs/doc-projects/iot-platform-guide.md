# 小米 IoT 开发者平台接入指南

本文介绍小米 IoT 开发者平台的产品接入流程，帮助开发者了解从创建产品到申请上线的大致步骤。

文中以一种典型的“标准模组 + MCU”接入方式作为示例，串联说明产品配置、功能定义、固件设置、模组联调、测试验证和上线申请等环节。

> 本文中的产品名称、模组型号、参数和测试结果仅作为流程参考，实际项目请以平台生成的信息和真实硬件方案为准。

---

## 1. 流程介绍

小米 IoT 平台接入可以理解为：先在平台上创建一个“产品定义”，再让真实设备按照这个产品定义完成配网、通信、控制、上报、测试和上线。

本文示例采用较典型的“标准模组 + MCU”接入方式：

- 标准模组负责联网、配网、云端通信和串口协议转发。
- MCU 负责按键、继电器、状态维护和具体业务逻辑。
- 米家 App 负责设备添加、设备控制和状态展示。
- 小米 IoT 平台负责产品配置、功能定义、固件版本、测试认证和上线管理。

整体流程如下：

| 阶段 | 开发者需要完成的事情 | 阶段产出 |
| --- | --- | --- |
| 前期准备 | 准备账号、权限、硬件、软件和网络环境 | 可以创建产品并开始调试 |
| 产品配置 | 创建产品，填写基础配置，选择品类模板 | 平台生成产品信息和功能定义 |
| 固件配置 | 选择开发模式、模组型号和 OTA 方式 | 明确设备接入方案 |
| 模组联调 | 初始化模组，完成米家 App 配网 | 设备可以接入米家和云端 |
| 功能验证 | 使用串口工具或 MCU 响应下发命令 | App 控制和设备状态同步正常 |
| 上线准备 | 完成高阶配置、固件版本、自测认证 | 具备提交上线的基础条件 |
| 申请上线 | 提交平台审核并根据反馈修改 | 产品正式进入上线流程 |

> 📸 截图占位：小米 IoT 接入流程图或平台产品详情总览页面。

## 2. 前提条件

开始接入前，开发者建议先按下面顺序检查准备情况，避免后续在产品创建、串口联调或上线申请阶段被权限和工具问题卡住。

1. 准备账号与权限
   - 注册并登录小米 IoT 开发者平台账号。
   - 加入企业开发组，确认账号具备产品创建、产品配置、固件开发等权限。
   - 提前添加品牌信息，后续产品上线和包装材料审核会用到。
   - 米家 App 使用同一开发者账号登录，或将测试账号加入开发组 / 白名单。
   - 如果页面按钮不可点击，优先检查账号权限，不要先怀疑设备或串口工具。

2. 准备硬件设备
   - 标准联网模组或开发板，例如本文示例使用的 `MHCWB8G-B`。
   - 支持数据传输的 Type-C 数据线，不能只用充电线。
   - Windows 电脑，用于识别串口和运行串口工具。
   - 手机一台，用于安装米家 App、添加设备和验证控制效果。
   - 如果只是验证平台流程，可以先用串口工具模拟 MCU；如果要做真实产品，还需要 MCU、继电器、实体按键和电源电路等硬件。

3. 安装软件与工具
   - 浏览器：访问小米 IoT 开发者平台。
   - 米家 App：用于设备配网、绑定和控制验证。
   - 串口工具：本文使用 `XCOM V2.6`，也可以使用同类串口调试工具。
   - USB 串口驱动：如果电脑无法识别 COM 口，需要安装对应驱动。
   - MCU 开发环境：真实产品开发阶段才需要，用于编写、编译和打包 MCU 固件。

4. 确认网络与手机环境
   - 手机开启蓝牙。
   - 手机连接 `2.4GHz Wi-Fi`，首次配网不建议使用 5GHz Wi-Fi。
   - 不建议使用需要网页认证的校园网、公司访客网或限制较多的公共网络。
   - 路由器如果开启 AP 隔离、MAC 白名单等限制，可能影响设备联网。

5. 整理产品基础信息

| 信息 | 说明 |
| --- | --- |
| 产品名称 | 用于平台和米家 App 展示 |
| 产品品类 | 决定可选择的品类模板和默认功能 |
| 品牌名称 | 需要与平台品牌、包装材料保持一致 |
| 联网方式 | 例如 Wi-Fi、BLE、Wi-Fi + BLE |
| 配网方式 | 决定米家 App 添加设备时的引导流程 |
| 模组型号 | 固件设置时需要选择对应模组 |
| `Model` / `PID` | 平台创建产品后生成，后续需要写入模组 |
| MCU 版本号 | 用于固件版本管理和 OTA 测试 |

> 本文示例中，真实 MCU 尚未接入，前期通过 XCOM 手动模拟 MCU 行为，用来验证小米 IoT 平台、模组、米家 App 和串口通信的主流程。

## 3. 操作步骤

下面按照平台接入顺序展开操作。实际项目中，不同品类和不同模组的页面字段可能略有差异，但整体流程基本一致。

### 3.1 创建产品

进入平台：

```text
产品管理 → 新建产品
```

按实际产品填写基础信息：

| 字段 | 说明 |
| --- | --- |
| 产品类型 | 选择与实际产品最接近的品类 |
| 产品名称 | 填写后会在平台和米家 App 中展示 |
| 品牌名称 | 选择已添加并通过审核的品牌 |
| 联网方式 | 按产品方案选择，例如 Wi-Fi + BLE |
| 配网方式 | 按模组和产品能力选择 |
| 产品参数 | 按真实硬件能力填写，例如按键数量、开关类型等 |

创建完成后，平台会生成产品对应的 `Model` 和 `PID`。这两个信息后续需要写入模组，用于让模组识别自己属于哪个平台产品。

> 📸 截图占位：产品创建页面，展示产品类型、联网方式、产品名称等字段。

### 3.2 完成基础配置

进入：

```text
产品详情 → 基础配置
```

重点确认以下内容：

| 配置项 | 说明 |
| --- | --- |
| 接入渠道 | 通常选择米家 App |
| 产品名称 | 与产品创建时保持一致 |
| 品牌名称 | 与企业品牌和包装材料保持一致 |
| 展示分类 | 决定米家 App 中的品类展示 |
| 产品配网方式 | 需要与模组和实际配网方式一致 |
| 产品离线提示 | 建议配置，方便用户理解设备离线原因 |
| 产品参数 | 与真实硬件能力一致 |

保存后，基础配置状态应变为“已完成”。

> 📸 截图占位：基础配置完成页，展示产品状态和配网方式。

### 3.3 使用品类模板定义功能

进入：

```text
产品详情 → 功能定义
```

新手接入建议优先使用平台提供的品类模板，而不是一开始就自定义功能。使用品类模板的好处是：

- 平台会自动生成该品类常用的服务、属性和事件。
- 米家 App 标准页面可以直接识别并展示常用功能。
- 小爱同学、自动化、定时等能力更容易适配。
- 后续审核时更符合平台的标准产品体验。

以双键开关为例，联调时重点关注左右两路开关属性：

| 功能 | 示例标识 | 说明 |
| --- | --- | --- |
| 左键开关 | `2 1` | 控制或查询左路开关状态 |
| 右键开关 | `3 1` | 控制或查询右路开关状态 |

> 具体标识以平台功能定义页面为准，本文示例仅用于说明串口联调过程。

> 📸 截图占位：功能定义中选择品类模板的页面。

### 3.4 完成固件设置

进入：

```text
产品详情 → 固件开发 → 固件设置
```

按实际方案选择开发模式和模组信息。本文示例使用“标准模组 + MCU”方式：

| 字段 | 示例选择 |
| --- | --- |
| 操作系统 | RTOS |
| 开发模式 | 小米标准模组 + MCU |
| OTA 方式 | MCU 固件 OTA |
| 安全认证 | 按项目实际要求选择 |
| 模组型号 | `MHCWB8G-B` |
| 芯片 | `BL616C` |

在这种模式下，标准模组主要负责联网和协议通信，MCU 负责真实硬件控制。也就是说，模组并不会自动完成所有开关业务逻辑，真实产品仍需要 MCU 或等效控制电路。

> 📸 截图占位：固件设置完成页，展示开发模式、模组型号和 OTA 方式。

### 3.5 准备串口环境并初始化模组

将模组或开发板通过 Type-C 数据线连接电脑，在设备管理器中查看串口：

```text
设备管理器 → 端口 (COM 和 LPT)
```

可能会出现两个串口：

| 串口类型 | 现象 |
| --- | --- |
| 日志口 | 持续输出 `miio_net`、`wifi scan` 等日志 |
| 命令口 | 可以发送 `get_down`、`model`、`ble_config` 等命令 |

XCOM 建议配置如下：

| 参数 | 值 |
| --- | --- |
| 波特率 | `115200` |
| 数据位 | `8` |
| 停止位 | `1` |
| 校验位 | `None` |
| 16 进制显示 | 不勾选 |
| 16 进制发送 | 不勾选 |
| 发送新行 | 勾选 |

在命令口依次发送初始化命令：

```text
set_log_level 0
model miot.switch.jing
ble_config set 39417 0001
mcu_version 0001
get_down
```

命令含义如下：

| 命令 | 作用 |
| --- | --- |
| `set_log_level 0` | 降低日志输出等级，减少干扰日志 |
| `model miot.switch.jing` | 写入平台生成的产品 `Model` |
| `ble_config set 39417 0001` | 写入 `PID` 和 BLE 配置版本 |
| `mcu_version 0001` | 设置当前 MCU 固件版本 |
| `get_down` | 获取模组下发给 MCU 的命令 |

> 实际项目中，`Model`、`PID` 和版本号需要替换为平台当前产品生成的信息。

> 📸 截图占位：设备管理器 COM 口、XCOM 串口参数、初始化命令返回 `ok`。

### 3.6 使用米家 App 配网

配网前确认手机蓝牙已开启，并连接 `2.4GHz Wi-Fi`。

操作步骤：

1. 打开米家 App。
2. 使用开发者测试账号登录。
3. 点击右上角 `+` 添加设备。
4. 选择或搜索当前测试产品。
5. 按页面提示输入 Wi-Fi 信息。
6. 等待设备添加完成。

配网成功后，在 XCOM 中发送：

```text
get_down
```

如果返回类似内容，说明设备已连接云端：

```text
down MIIO_net_change cloud
```

> 📸 截图占位：米家 App 添加设备成功页、XCOM 中显示设备联网状态。

### 3.7 使用 XCOM 模拟 MCU 联调

在没有真实 MCU 的阶段，可以使用 XCOM 手动模拟 MCU。联调逻辑如下：

```text
米家 App 控制设备
→ 标准模组通过串口下发命令
→ XCOM 手动回复执行结果
→ XCOM 手动上报属性变化
→ 米家 App 展示最新状态
```

常见下发、回复和上报关系如下：

| 场景 | 模组下发 | MCU / XCOM 需要做的事 |
| --- | --- | --- |
| 查询状态 | `down get_properties ...` | 回复当前属性值 |
| 控制开关 | `down set_properties ...` | 回复执行结果，并上报新状态 |
| 本地状态变化 | 无需等待 App 下发 | 主动上报 `properties_changed` |

查询左右键状态：

```text
down get_properties 2 1 3 1
```

左右都关时回复：

```text
result 2 1 0 false 3 1 0 false
```

左右都开时回复：

```text
result 2 1 0 true 3 1 0 true
```

左键打开：

```text
down set_properties 2 1 true
result 2 1 0
properties_changed 2 1 true
```

右键打开：

```text
down set_properties 3 1 true
result 3 1 0
properties_changed 3 1 true
```

全开：

```text
down set_properties 2 1 true 3 1 true
result 2 1 0 3 1 0
properties_changed 2 1 true 3 1 true
```

全关：

```text
down set_properties 2 1 false 3 1 false
result 2 1 0 3 1 0
properties_changed 2 1 false 3 1 false
```

本文示例已完成左键、右键、全开、全关、状态查询和状态上报验证。

> 📸 截图占位：XCOM 中完成左右键、全开、全关测试的记录。

### 3.8 完成高阶配置

基础联调完成后，进入：

```text
产品详情 → 高阶配置
```

上线前通常需要完成以下配置：

| 配置项 | 说明 |
| --- | --- |
| 配网引导 | 配置设备添加时的联网说明和初始化页面 |
| 使用帮助 | 配置米家 App 中展示的产品帮助文档 |
| 产品百科 | 配置说明书、购买链接、产品信息等 |
| 文案多语言 | 配置产品相关多语言文案 |
| 产品包装材料 | 上传包装、说明书、标签等审核材料 |

配网引导示例：

```text
1. 请确认设备已正常上电。
2. 请确认手机已开启蓝牙，并连接 2.4GHz Wi-Fi。
3. 如设备无法被发现，请将设备恢复出厂设置后重新添加。
```

使用帮助可以补充以下常见问题：

```text
问题：设备无法被米家 App 发现怎么办？
回答：请确认手机蓝牙已开启，并连接 2.4GHz Wi-Fi；请确认设备已上电并处于配网状态；如仍无法发现，请恢复出厂设置后重新添加。
```

> 📸 截图占位：高阶配置列表、配网引导编辑页、使用帮助编辑页。

### 3.9 创建固件版本与 OTA 测试

进入：

```text
产品详情 → 固件开发 → 固件版本管理
```

在“标准模组 + MCU”模式下，平台中上传的固件包通常是外部 MCU 的业务固件包，不是标准模组本身的固件。真实产品中，MCU 固件需要实现按键读取、继电器控制、状态维护、串口通信和 OTA 升级等能力。

固件版本操作流程：

1. 点击“新建固件版本”。
2. 选择固件类型，例如 `MCU 固件`。
3. 填写固件版本号和版本说明。
4. 上传 MCU 编译生成的固件包。
5. 保存并提交测试。
6. 将固件推送给测试设备。
7. 在米家 App 中验证升级流程。
8. 升级完成后重新测试核心功能。

如果当前只是使用 XCOM 模拟 MCU，可以在项目记录中说明：当前阶段已完成平台链路验证，真实 MCU 固件包需要在 MCU 代码开发完成后生成并上传。

> 📸 截图占位：固件版本管理页面、新建固件版本弹窗、米家 App 固件升级页面。

### 3.10 完成自测与认证预约

进入：

```text
产品详情 → 自测与预约
```

双键开关建议至少覆盖以下测试项：

| 测试项 | 验证目标 |
| --- | --- |
| 配网测试 | 设备可以被米家 App 发现并成功绑定 |
| 左键控制 | App 控制左键开关，设备状态正确变化 |
| 右键控制 | App 控制右键开关，设备状态正确变化 |
| 全开全关 | 同时控制两路开关，状态一致 |
| 状态查询 | App 进入页面后可以获取当前左右键状态 |
| 本地按键 | 按实体按键后，App 状态同步变化 |
| 断网恢复 | 路由器断开再恢复后，设备能重新上线 |
| 断电恢复 | 设备重新上电后，状态和联网行为符合设计 |
| OTA 升级 | 固件升级过程稳定，升级后功能正常 |
| 自动化联动 | 与米家自动化、小爱同学等能力配合正常 |

没有真实 MCU 和继电器硬件时，可以先完成平台链路类测试；本地按键、继电器通断、断电恢复和真实 OTA 需要等硬件和 MCU 固件完成后再测试。

> 📸 截图占位：自测与预约页面、测试文档下载入口、测试结果记录。

### 3.11 提交产品上线申请

进入：

```text
产品详情 → 产品申请上线
```

提交前建议按以下清单确认：

| 模块 | 检查内容 |
| --- | --- |
| 基础配置 | 产品名称、型号、品牌、品类、销售区域已填写 |
| 功能定义 | 功能与实际硬件能力一致 |
| 固件开发 | 模组配置、MCU 版本、OTA 流程已完成 |
| 高阶配置 | 配网引导、使用帮助、百科、包装材料已配置 |
| App 体验 | 设备卡片、控制页、状态展示符合预期 |
| 小爱能力 | 常用语音控制测试通过 |
| 自动化 | 可作为触发条件或执行动作正常使用 |
| 包装材料 | 说明书、二维码、型号信息与平台一致 |
| 自测认证 | 自测通过，必要时完成认证预约 |

上线申请通常会经历：

```text
提交上线申请 → 平台资料审核 → 产品体验 / 功能审核 → 问题修改 → 重新提交 → 审核通过 → 正式上线
```

常见退回原因包括：产品名称和包装材料不一致、功能定义与实际硬件能力不一致、配网引导不清楚、OTA 测试不完整、使用帮助缺少复位或离线说明等。

> 📸 截图占位：产品申请上线页面、上线前待完成事项、审核状态页面。

1. 完善文档展示
   - 补充平台关键页面截图。
   - 补充设备管理器和 XCOM 调试截图。
   - 补充米家 App 配网、控制和状态同步截图。
   - 将示例参数统一标注为“仅供参考”，避免被误认为固定配置。

2. 推进真实硬件开发
   - 选择 MCU 或带 MCU 的成品方案板。
   - 将 XCOM 中手动发送的逻辑写入 MCU 程序。
   - 接入实体按键、继电器和状态指示灯。
   - 编译生成 MCU 固件包。
   - 在平台创建固件版本并验证 OTA。
   - 完成自测、认证预约和上线申请。
