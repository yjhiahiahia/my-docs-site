这个文档站是我给自己做的一个作品集。最开始的想法很简单：面试或者沟通作品时，不想再发一堆零散文件，而是希望有一个地方能集中展示我写过的技术文档、建站过程和一些工作思考。

## 为什么要搭一个站

技术文档不像设计作品，可以直接放几张图就看出效果。文档更需要上下文：它写给谁看、解决什么问题、结构怎么安排、读者看完能不能继续往下走。

之前我试过几种方式，但都不太合适：

| 方式 | 问题 |
| --- | --- |
| Word / PDF | 文件太分散，别人不一定愿意一个个下载打开 |
| 在线文档 | 有时需要权限，展示起来不够稳定 |
| GitHub README | 适合放项目说明，不适合放多篇长文档 |

所以最后决定搭一个文档站。这样一来，别人打开链接就能看到首页、导览页、文档作品和建站手记，阅读路径会更清楚。

## 我主要做了什么

我不是程序员，所以这个站不是为了证明自己会写代码。对我来说，更重要的是把内容组织好，把入口放清楚，让读者知道应该先看哪里。

整个过程里，我主要做了这些事：

1. **确定站点用途**：这个站是技术文档作品集，不是个人简历，也不是博客合集。
2. **规划内容结构**：把内容分成“文档作品”和“建站手记”两大块。
3. **搭起基础站点**：用 Docusaurus 搭了一个可以放 Markdown 文档的网站。
4. **整理首页和导览页**：让第一次进来的人能快速判断自己该看哪篇。
5. **补充质量检查流程**：用格式检查和 AI 评分，帮我发现文档哪里还不够清楚。
6. **部署到 GitHub Pages**：把站点发布成一个可以直接访问的网页。

其中比较复杂的配置和页面代码，我基本是让 AI 帮我处理。我负责决定“要做成什么样”“内容怎么排”“哪些地方要删掉或重写”。

## 站点现在怎么组织

目前站点主要分成四个入口：

| 入口 | 放什么 | 读者能看到什么 |
| --- | --- | --- |
| 首页 | 站点整体介绍和重点卡片 | 快速知道这个站是做什么的 |
| 站点导览 | 所有内容的阅读路径 | 按目标选择先看哪篇文档 |
| 文档作品 | IoT 概念、接入指南、API 文档 | 展示不同类型技术文档的写法 |
| 建站手记 | 建站过程、质量评分、更新流程、工作思考 | 展示我是怎么维护这个站的 |

文件大致是这样放的：

```text
docs/
├── intro.md              # 站点导览
├── iot-overview.md       # IoT 核心概念
├── iot-platform-guide.md # 小米 IoT 平台接入指南
└── api-docs.md           # IoT 平台 API 文档

blog/
├── 2026-07-01-docs-as-code.md       # 建站过程
├── 2026-07-10-doc-score.md          # 文档质量评估助手
├── 2026-07-14-update-workflow.md    # 文档站更新流程
└── 2026-07-14-iot-doc-thoughts.md   # IoT 文档调研思考
```

这个结构不是一开始就完全想好的，是边写边调整出来的。比如一开始我把“文档调研思考”放在文档作品里，后来发现它更像工作复盘，就挪到了建站手记里。

## 首页直接放几个入口

首页没有放很多自我介绍，而是直接放几个入口，让读者一进来就能选方向：

- 想看文档作品，可以点到作品集。
- 想看建站过程，可以点到建站手记。
- 想快速判断内容，可以看首页下面的卡片。

我希望首页解决的是“来了以后先看哪里”的问题，而不是把所有内容都堆在第一页。

## 导览页怎么想的

导览页是这个站里我比较重视的一页。因为作品集最怕的问题是：内容不少，但读者不知道从哪里开始。

所以我在导览页里放了几类信息：

| 内容 | 作用 |
| --- | --- |
| 快速选择 | 让读者按目标直接进入对应文档 |
| 站点地图 | 用图看清楚内容之间的关系 |
| 推荐阅读路径 | 告诉不同读者应该先看哪几篇 |
| 文档详情表 | 说明每篇文档适合谁、讲什么、需要多久 |

这样读者不用每篇都点开试，可以先判断哪篇最符合自己的需求。

## 文档作品怎么选的

我没有把所有写过的东西都放上来，而是挑了几种能体现不同能力的文档：

| 文档 | 展示重点 |
| --- | --- |
| IoT 核心概念 | 能不能把抽象概念讲清楚 |
| 小米 IoT 平台接入指南 | 能不能把长流程拆成可执行步骤 |
| IoT 平台 API 文档 | 能不能把接口说明写得清楚、可查 |

这些文档不一定都是“最终完美版”，但它们能展示我怎么处理概念、流程、接口、截图、表格、排障和下一步链接。

## 建站手记放什么

建站手记更像过程记录。我会把和这个站有关的思考放在这里，比如：

- 为什么要搭这个站。
- 文档质量评分怎么接进来。
- 平时怎么更新文档。
- 看完竞品文档后有哪些工作思考。

这些内容不完全是正式文档，更像“我做这件事时怎么想、怎么改、怎么复盘”。

## 小结

搭这个站最大的收获，不是学会了某个工具，而是重新想清楚了作品集应该怎么呈现。

对我来说，一个好的文档站不只是“把文章放上去”，而是要让读者一进来就知道：这里有什么、先看什么、每篇内容能证明什么能力。这个站后面还会继续改，但现在它已经能比较完整地展示我做过的文档和我对文档工作的理解。
